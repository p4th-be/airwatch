INTRODUCTION
====================================================================

Pour ce devoir intégré, j’ai choisi de travailler sur un site consacré à la qualité de l’air en Europe. Le sujet m’a semblé intéressant parce qu’il touche à la santé, à l’environnement et aux données publiques. Aujourd’hui, la pollution atmosphérique est un problème réel qui concerne beaucoup de personnes, mais les données restent souvent dispersées, techniques ou difficiles à comprendre pour un public non spécialiste. Mon objectif a donc été de créer un site capable de rendre ces informations plus lisibles, plus visuelles et plus utiles.

Le public cible de mon site est assez large. Je vise d’abord les citoyens qui veulent comprendre la qualité de l’air dans certaines villes, mais aussi les élèves, les enseignants et les personnes qui s’intéressent aux questions de santé publique et d’écologie. Je voulais que le site soit informatif, visuel et relativement simple à parcourir, sans tomber dans un design trop vide ou trop scolaire.

L’objectif principal du site est de centraliser et présenter des données liées à la qualité de l’air, aux polluants, aux stations de mesure, aux impacts sanitaires et aux sources de données utilisées. Je voulais aussi aller plus loin qu’un simple site vitrine, en ajoutant une partie dynamique avec une gestion administrateur, des formulaires, des évaluations utilisateurs et une publication accessible en ligne.

Au tout début du projet, j’ai commencé en mode Python simple, avec des scripts pour comprendre les données, faire des essais rapides et vérifier la logique de traitement. Cette étape m’a servi de base pour tester mes idées sans complexité de framework. Ensuite, quand le projet est devenu plus grand (pages multiples, authentification, base relationnelle, administration, publication), je suis passé sur Django pour avoir une architecture propre et durable.

Ce sujet se prête très bien à l’utilisation de données open data parce qu’il existe des plateformes publiques qui fournissent de vraies mesures environnementales exploitables. Dans mon projet, je me suis appuyé surtout sur des données issues d’OpenAQ et sur des références liées à la qualité de l’air en Europe. Ce choix me permettait de relier un besoin concret à une base de données réelle et de construire un projet crédible, documentable et évolutif.

PARTIE 2 – SECURISATION ET OPTIMISATION DU SITE WEB
====================================================================

A. Diagnostic du site

Au début, plusieurs points faibles ont été identifiés :
- une transition encore incomplète entre site informatif et application dynamique ;
- une dépendance forte à la configuration serveur ;
- un risque d’écrasement des données si l’admin modifiait une donnée ensuite remplacée par une mise à jour automatique ;
- un problème de gestion des fichiers statiques après la mise en place de nginx ;
- des difficultés de cohérence entre les domaines publics et les certificats HTTPS.

B. Améliorations de performance

J’ai mis en place plusieurs améliorations :
- utilisation de Whitenoise pour une meilleure gestion des fichiers statiques ;
- organisation claire des dossiers `static` et `templates` ;
- limitation de certaines requêtes grâce à l’usage de `select_related` dans plusieurs vues ;
- centralisation des styles et scripts.

Ces choix améliorent le site car ils rendent le chargement plus propre, limitent certaines erreurs de ressources et simplifient la maintenance.

C. Améliorations de sécurité

J’ai identifié plusieurs risques :
- exposition des actions sensibles si n’importe quel utilisateur pouvait modifier les données ;
- écrasement de données éditées manuellement ;
- erreurs de configuration liées à l’accès admin ;
- certificats HTTPS incorrects selon le domaine utilisé.

Mes réponses techniques :
- mise en place de l’authentification Django ;
- restriction des pages d’administration avec `login_required` et `user_passes_test(is_admin)` ;
- utilisation de formulaires Django validés côté serveur ;
- ajout du champ `manually_edited` pour protéger les données modifiées à la main ;
- séparation claire entre visiteurs et administrateurs.

D. Comparaison avant / après

Avant :
- site plus proche d’une vitrine ;
- données moins contrôlées ;
- risque d’écrasement des modifications ;
- configuration serveur plus fragile.

Après :
- site structuré comme une application ;
- contenu administrable ;
- protection des données éditées manuellement ;
- meilleure logique d’accès ;
- publication publique stabilisée.

PARTIE 3 – ANALYSE DE LA TRANSFORMATION DU SITE STATIQUE EN SITE DYNAMIQUE

A. Choix technique du backend

J’ai choisi Django comme technologie backend. Ce choix est adapté parce que Django fournit directement un système complet avec modèles, vues, templates, formulaires, authentification, administration et sécurité de base. Pour un projet scolaire mais concret, c’était une solution cohérente, rapide à mettre en place et assez professionnelle.

Avantages :
- structure claire ;
- ORM puissant ;
- admin intégré ;
- système d’authentification déjà prêt ;
- gestion facilitée des formulaires et validations.

Limites :
- plus lourd qu’un petit site statique ;
- demande une vraie configuration serveur ;
- certaines parties peuvent sembler complexes au début.

B. Ce qui devient dynamique

Restent principalement statiques :
- la structure générale du site ;
- certains textes explicatifs ;
- la navigation principale.

Deviennent dynamiques :
- la liste des villes ;
- les stations ;
- les mesures ;
- les polluants ;
- les impacts ;
- les sources ;
- les évaluations utilisateurs ;
- l’affichage du dashboard admin.

La différence visible pour l’utilisateur est importante : le contenu n’est plus seulement écrit à la main dans du HTML, il est récupéré depuis une base et peut évoluer sans refaire la page entière.

C. Gestion des données

Les données viennent de plusieurs sources :
- données open data ;
- contenus saisis ou corrigés par l’administrateur ;
- évaluations laissées par les visiteurs.

Elles sont stockées dans la base de données Django. Dans le projet, plusieurs modèles structurent les informations :
- `Pollutant`
- `Country`
- `City`
- `MeasurementStation`
- `AirQualityData`
- `HealthImpact`
- `DataSource`
- `Evaluation`

Les données sont lues via les vues Django, puis affichées dans les templates. Elles sont modifiées soit par des formulaires spécifiques, soit par le panneau d’administration.

D. Gestion des images dynamiques

Dans mon projet actuel, les images restent encore assez simples, mais la logique prévue est la suivante :
- associer certaines images aux villes ou contenus ;
- stocker les chemins ou URLs dans la base ;
- vérifier les liens avant affichage ;
- prévoir qu’un administrateur puisse modifier les contenus sans casser les pages.

La présence du champ `image_url` dans le modèle `City` montre déjà cette logique de liaison entre contenu et média.

- base de données : base relationnelle Django ;
- serveur : Ubuntu + nginx + application Django ;

Pages concernées par le passage au dynamique :

Visiteur -> pages publiques / évaluations -> base de données

F. POC et difficultés techniques

Une vraie difficulté du projet a été la coexistence entre données automatiques et données modifiées à la main. Si je laissais l’actualisation automatique écraser toutes les données, l’administrateur risquait de perdre ses corrections.

J’ai donc réalisé un mini POC logique puis une vraie implémentation :
- ajout d’un champ `manually_edited` sur plusieurs modèles ;
- activation automatique de ce champ quand l’admin sauvegarde ;
- adaptation de la commande `fetch_openaq_data.py` pour ne pas remplacer les données marquées comme modifiées manuellement.

Ce test m’a appris qu’une application dynamique ne consiste pas seulement à afficher une base de données. Il faut aussi penser aux conflits entre automatisation et intervention humaine.

CHRONOLOGIE DETAILLEE DU PROJET (TRAVAIL REEL)
====================================================================

Cette section reprend de manière chronologique les étapes importantes du projet, avec les problèmes rencontrés et les corrections appliquées. Le but est de montrer un vrai parcours de développement, avec des choix, des erreurs, des ajustements et des résultats concrets.

1) Démarrage du projet et cadrage du sujet

Au début, j’ai défini un objectif simple : créer un site qui explique la qualité de l’air en Europe avec des données ouvertes. J’ai rapidement compris que ce sujet me permettrait de faire un vrai projet complet : front-end, back-end, base de données, administration et publication.

Le premier travail a été de structurer les idées en rubriques claires : accueil, données, carte, impacts, sources. J’ai ensuite défini une progression en deux temps : d’abord une version de type statique/informative, puis une version dynamique administrable.

Avant d’entrer directement dans Django, j’ai donc utilisé Python seul pour plusieurs essais techniques : scripts de récupération de données, vérifications de format, et automatisation de certaines tâches serveur. Cette phase “Python d’abord” m’a aidé à comprendre la donnée et les problèmes concrets. Une fois les besoins clarifiés, j’ai migré vers Django pour organiser le code proprement et ajouter toutes les fonctionnalités web attendues.

2) Mise en place de la structure Django

J’ai installé le projet avec Django et créé l’application principale. Très vite, j’ai séparé la logique en plusieurs couches :
- modèles pour structurer les données ;
- vues pour la logique métier ;
- templates pour l’affichage ;
- fichiers statiques pour le style et les scripts.

J’ai aussi préparé une arborescence propre pour éviter de mélanger toutes les responsabilités dans un seul fichier. Cette étape m’a fait gagner du temps plus tard, surtout quand il a fallu corriger des bugs sans casser d’autres parties.

3) Conception des modèles de données

J’ai défini les modèles autour de la logique qualité de l’air :
- `Pollutant` ;
- `Country` ;
- `City` ;
- `MeasurementStation` ;
- `AirQualityData` ;
- `HealthImpact` ;
- `DataSource` ;
- `Evaluation`.

Le plus important était de bien poser les relations : une ville appartient à un pays, une station appartient à une ville, une mesure relie une station et un polluant. Cette relationnalité a permis d’obtenir des pages utiles (exemple : détail d’une ville avec ses mesures récentes).

4) Développement des vues publiques

J’ai ensuite construit les vues publiques :
- page d’accueil avec statistiques et dernières mesures ;
- page de données avec agrégations (moyennes, min, max) ;
- page carte ;
- page impacts sanitaires ;
- page sources ;
- page de détail d’une ville.

Cette étape m’a permis de valider que les données étaient bien récupérées et affichées. C’était aussi un premier test de cohérence entre les modèles et les templates.

5) Mise en place de l’authentification et de la séparation des rôles

Une priorité du sprint dynamique était de différencier un visiteur d’un administrateur. J’ai ajouté :
- une vue de connexion ;
- une vue de déconnexion ;
- des restrictions d’accès avec `login_required` ;
- un contrôle admin avec `user_passes_test(is_admin)`.

Avec ça, les pages sensibles (dashboard et CRUD) sont protégées et non accessibles au public.

6) Création du tableau de bord administrateur et des CRUD

J’ai créé un dashboard personnalisé qui centralise :
- des compteurs utiles ;
- les listes de données ;
- les actions principales.

Puis j’ai ajouté les opérations CRUD sur plusieurs entités :
- villes ;
- mesures ;
- polluants.

L’intérêt concret : après connexion admin, je peux ajouter, modifier et supprimer des informations, et voir immédiatement le résultat dans les pages publiques.

7) Ajout des évaluations utilisateurs

J’ai développé une page d’évaluations avec formulaire. Les utilisateurs peuvent laisser une note et un commentaire. Les données sont stockées dans le modèle `Evaluation`. Le système inclut aussi une logique de validation/modération via l’admin.

Ce point est important car il répond à la demande d’interaction utilisateur dans le sprint 1.

8) Problème majeur : conflit entre mise à jour automatique et édition manuelle

Quand des données sont alimentées automatiquement, il existe un risque réel : écraser des corrections faites à la main par l’admin. C’est un problème typique dans les projets data-driven.

Pour éviter ça, j’ai mis en place une solution robuste :
- ajout d’un champ `manually_edited` dans les modèles concernés ;
- ajout d’un mixin admin pour marquer automatiquement une donnée comme éditée manuellement ;
- adaptation de la commande de récupération OpenAQ pour ignorer les enregistrements protégés.

Résultat : les données corrigées en admin ne sont plus remplacées automatiquement.

9) Déploiement serveur et difficultés réelles

Le déploiement a été fait sur Ubuntu avec nginx en reverse proxy. Django tourne en backend, nginx sert d’entrée publique.

J’ai rencontré plusieurs problèmes techniques réels :
- certaines commandes distantes bloquaient ;
- des erreurs de configuration de service ;
- des incohérences sur les ports ;
- des problèmes de certificat selon le domaine utilisé.

Ces problèmes m’ont obligé à avancer en mode diagnostic/correction, avec vérifications successives plutôt qu’une seule commande magique.

10) Problème CSS en production

À un moment, le site s’affichait sans style (page “blanche” côté design). La cause était liée à la manière dont nginx servait les fichiers statiques, en conflit avec la logique Whitenoise.

Correction appliquée :
- ajustement de la configuration nginx ;
- vérification des routes statiques ;
- test direct de chargement de `style.css`.

Après correction, le CSS a été correctement servi.

11) HTTPS et certificats

J’ai travaillé sur HTTPS pour sécuriser l’accès public. Le certificat Let’s Encrypt a été obtenu pour `airwatch102.duckdns.org` (domaine contrôlé). Pour `p4thweather.duckdns.org`, le certificat reconnu n’était pas possible tant que le domaine restait attaché à un autre compte DuckDNS.

Concrètement :
- HTTPS fonctionnel et reconnu sur le domaine contrôlé ;
- avertissement navigateur si accès via un domaine qui ne correspond pas au certificat.

Cette partie m’a appris qu’en sécurité web, la configuration technique seule ne suffit pas : il faut aussi maîtriser la propriété DNS du domaine.

12) Versionnement et publication GitHub

Le projet a été poussé sur GitHub dans l’organisation prévue. Pendant cette étape, j’ai rencontré la protection GitHub contre les secrets : certains scripts temporaires contenaient des tokens en clair, ce qui a bloqué le push.

Actions correctives :
- suppression des scripts sensibles de l’index git ;
- mise à jour du `.gitignore` ;
- nettoyage et commit propre ;
- nouveau push validé.

Résultat : dépôt publié proprement sans exposer de secret.

13) Tests fonctionnels finaux

J’ai vérifié les points suivants :
- accès public au site ;
- navigation entre les pages ;
- accès login/logout ;
- accès dashboard admin protégé ;
- création/modification/suppression de données ;
- dépôt d’évaluations ;
- affichage des statistiques principales ;
- état du serveur nginx + backend Django.

14) Bilan personnel de la démarche

Le point le plus marquant de ce projet est que j’ai travaillé sur un enchaînement complet : idée -> structure -> code -> données -> admin -> serveur -> publication.

Ce n’est pas un simple mini-site de démonstration. J’ai dû prendre des décisions techniques, corriger des erreurs réelles, adapter des solutions à mon environnement et vérifier chaque étape.

Ce travail m’a aussi appris à documenter ce que je fais, pas seulement à coder. La traçabilité, les captures, les justifications et les choix expliqués sont devenus une partie importante du projet, pas juste un ajout de fin.
